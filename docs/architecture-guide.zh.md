# 架构解读（中文）

## 1. 定位与边界

Wui 的定位是 Agent 运行时，而不是 Agent 策略框架。

这句话在工程上意味着：
- 框架负责把模型决策可靠地执行出来
- 框架不负责替模型规划任务流程
- 框架不把某个产品形态固化进核心抽象

因此，Wui 的核心职责集中在执行语义：流式输出、工具调度、权限控制、
上下文压缩、会话与恢复。

## 2. 分层结构：词汇层、执行层、能力扩展层

仓库采用单向依赖的分层结构：

- `wui-core`：跨运行时共享的稳定词汇（`Tool`、`Provider`、消息与事件等）
- `wui`：实际执行引擎与公开 API（运行循环、权限、压缩、checkpoint 等）
- 扩展 crate：`wui-memory`、`wui-mcp`、`wui-observe`、`wui-spawn`、`wui-skills`、`wui-eval`

这套分层带来三个直接结果：
- 自定义工具/Provider 可以只依赖 `wui-core`，减少耦合
- 扩展能力可演进，不必频繁改动核心词汇
- 运行时策略不“污染”基础 trait 设计

## 3. 主循环：执行语义的中轴

Wui 的核心行为集中在 `crates/wui/src/runtime/run`。

单次迭代可以概括为：

1. 进入回合前检查上下文压力，必要时压缩
2. 基于系统提示词、历史和工具目录构建 `ChatRequest`
3. 调用 Provider 流接口（带重试策略）
4. 消费流事件（eager dispatch）：
- 文本增量直接向外发出
- 工具输入闭合（`ToolUseEnd`）后立即执行 pre-hook + 权限判定
- 可立即决定的调用（允许/拒绝）当场处理
- 需要人工审批的调用进入 deferred 队列
- 遇到 `MessageEnd` 结束本轮模型流阶段
5. 对 deferred 工具并发做 HITL 授权并分发执行
6. 汇总执行结果并运行 post-tool hooks
7. 按提交顺序把工具结果回写到历史
8. 写入系统提醒类注入消息
9. 判断停止条件，决定继续或结束

这里有两个关键工程点：
- 事件通道有背压，不做无限缓存
- 工具可并发执行，但历史写回顺序可预测

这保证了系统既有吞吐能力，也有可解释性。

## 4. 事件模型：Provider 协议与应用协议解耦

Wui 区分两层事件：

- `StreamEvent`：Provider 层原始流协议
- `AgentEvent`：对外暴露的运行时语义事件

这样设计的好处是：
- Provider 适配层可以处理不同厂商流协议差异
- 上层应用拿到的事件保持稳定，不被底层细节牵动

换句话说，Provider 可以替换，应用事件处理逻辑不必重写。

## 5. 工具系统：按“调用实例”决定语义

`Tool` trait 保持最小必要面：名称、描述、输入 schema、异步调用。

运行时行为由两组元信息补充：
- `ToolMeta`：并发性、只读/破坏性、是否必须交互、权限作用域键
- `ExecutorHints`：超时、输出截断、重试等执行器参数

一个重要设计是“按调用实例决定并发语义”。
例如同一个 shell 工具，`ls` 可以并发，`rm` 可以串行或更严格授权。

另外，Wui 明确保证：
即使工具完成顺序不同，写入会话历史时仍按原提交顺序回写。

## 6. 权限链路与 HITL

权限不是外围插件，而是主循环的一部分。

判定顺序遵循先硬约束、再会话记忆、再模式行为：
1. pre-tool hook
2. 结构性检查（如需要交互但当前无交互能力）
3. 静态 deny
4. 会话 deny
5. 静态 allow
6. 会话 allow
7. `Ask/Auto/Readonly` 等模式决策

HITL 触发时机也很关键：
只有当模型本轮输出结束后，才发出 `AgentEvent::Control` 让人类审批。
但这只针对需要交互审批的调用；无需 HITL 的调用可以在流式阶段提前启动。

这让使用者可以先看到完整意图，再做授权决定，避免“半句打断”。

## 7. 上下文压力与压缩策略

Wui 把上下文有限视为常态，不把它当异常边缘案例。

压缩链路采用分层渐进策略：
- 轻量裁剪
- 结构性折叠旧历史
- 必要时摘要化

只有前一层不足时才升级到后一层。
如果仍无法缓解，会返回明确的 overflow 停止原因，
并支持 `on_context_overflow` 回调做自定义恢复。

这个策略减少了两类常见问题：
- 低价值历史长期占用 token 预算
- 超长请求直接失败且缺乏恢复抓手

## 8. 会话与 checkpoint 的恢复语义

Wui 把“历史持久化”和“授权记忆”分开处理：
- Session store 持久化的是消息历史
- `approve_always/deny_always` 这类授权记忆属于会话态，不等同于长期策略存储

Checkpoint 以“迭代边界”为单位保存，不做工具执行中的中间快照。
这使恢复点语义简单明确：从最近一次完整边界继续，而不是从不确定中间态恢复。

## 9. 扩展能力的架构位置

扩展 crate 对应不同能力域：
- `wui-memory`：记忆接口与参考实现
- `wui-mcp`：MCP 工具桥接与延迟发现
- `wui-observe`：时间线与 OTel 可观测性
- `wui-spawn`：子 agent 委派
- `wui-skills`：文件化 skill 发现
- `wui-eval`：MockProvider 与测试 harness

核心点在于“可选接入”。
你可以按需组合能力，而不是被强制绑定某一种产品范式。

## 10. 工程取舍与适配建议

当前架构优势：
- 边界清晰，核心与扩展职责明确
- 运行时不变量有文档化约束（`docs/runtime-invariants.md`）
- 测试与评估路径完整（`wui-eval` + integration tests）

需要注意的现实约束：
- 多个扩展 crate 仍处于 beta/stabilizing 阶段，API 可能继续收敛
- 能力完整意味着概念密度较高，新团队有学习成本
- 业务策略仍需在应用层落地，框架不会替你做产品级决策

## 11. 建议阅读顺序

给新贡献者的高效路径：

1. `README.md`：先建立项目边界和 crate 地图
2. `docs/philosophy.md`：理解设计法则与非目标
3. `docs/runtime-invariants.md`：明确系统保证的行为
4. `crates/wui/src/runtime/run/`：进入执行主循环实现
5. `examples/` 任意一条完整链路：把抽象映射到可运行代码

这个顺序基本对应 Wui 的设计方式：先原则，再语义，再实现。
